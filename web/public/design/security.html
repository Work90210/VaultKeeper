Security model

Every claim VaultKeeper makes can be checked independently. The database enforces append-only at the engine level. The custody chain is verifiable with a MIT-licensed clerk tool — without us. The crypto is open. The threat model is written down. This page is what's in it.

Threat model

The four attacks that keep tribunals up at night.

Most security tools defend against an opportunistic attacker. International justice needs something harder: defence against a hostile state, a compromised cloud, and a corrupt insider — simultaneously. VaultKeeper's architecture makes each of these attacks fail loudly instead of silently.

A1

Sanction-driven shut-off

A foreign government pressures a cloud to freeze your instance. Self-hosted deployment + AGPL source + public docker images means a tribunal relocates in hours, not quarters.

A2

Custody tampering by insider

A root DB admin tries to edit the custody log. Row-level security refuses the UPDATE at the engine. Even if they bypass it, hash chaining makes the next read fail verification.

A3

Witness identity exposure

Defence counsel should never see cleartext names. Application-layer AES-256-GCM means even a database dump contains only opaque blobs; pseudonyms are generated per role.

A4

Forced access under duress

An investigator is coerced into logging in. A duress passphrase (planned, v1.3) unlocks a decoy vault while silently flagging the session in a signed audit record.

Architecture

Three sealed layers, one provable chain.

Every request passes through the same three gates: who are you, what may you do, and does this action belong on the chain? Each gate writes its outcome into the append-only log.

Cryptographic spine

Hashes of hashes, all the way up.

Every custody event is hashed with SHA-256 and combined into a Merkle tree. The root is signed, timestamped under RFC 3161, and optionally anchored to a public blockchain. A defence expert can verify any single event with a ~2 KB proof — without trusting us, without being online.

  • SHA-256 + BLAKE3 — primary + defence-in-depth hash
  • Ed25519 — signing keys on a YubiHSM 2
  • RFC 3161 timestamps from eIDAS-registered TSAs
  • OpenTimestamps daily anchor to Bitcoin L1
  • AES-256-GCM envelope encryption for witness PII
